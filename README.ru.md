# execenv

`execenv` — небольшая Unix-oriented entrypoint-утилита для контейнеров.

Она подставляет значения переменных окружения в плейсхолдеры `${VAR}` в одном или нескольких файлах, а затем может заменить свой процесс другим процессом через `exec(2)`.

Это полезно, когда приложению нужен статический конфигурационный файл, но итоговые значения известны только в момент запуска контейнера.

## Мотивация

`execenv` в первую очередь рассчитан на контейнерные образы, где конфигурацию нужно передавать во время запуска, а не во время сборки.

Типичный пример — frontend или статическое приложение на основе distroless, один и тот же образ которого используется в нескольких окружениях: development, staging, production. Вместо пересборки образа под каждое окружение можно хранить шаблон внутри образа и рендерить итоговую конфигурацию при старте контейнера.

## Возможности

- Подстановка `${VAR}` из переменных окружения.
- Обработка одного или нескольких файлов за один запуск.
- Перезапись файлов на месте или рендер из input-файла в отдельный output-файл.
- Настраиваемое поведение при отсутствии переменной окружения.
- Запись через временный файл с заменой целевого файла только после успешного рендера.
- Опциональный переход в финальный процесс приложения через `exec(2)`.
- Основной сценарий использования — Linux/Unix контейнеры.

## Ограничения

> [!IMPORTANT]
> - Размер буфера чтения по умолчанию — 256 KiB.
> - Максимальная длина имени переменной в плейсхолдере — 4 KiB. Более длинные незавершённые плейсхолдеры сохраняются как обычный текст.
> - Output-файл наследует права доступа от input-файла.
> - Поддерживаются только Unix-like системы.

## Установка

Собрать из исходников:

```sh
cargo build --release
```

Оптимизированный бинарник будет доступен по пути:

```sh
target/release/execenv
```

Проект требует **Rust 1.91** или новее.

## Использование

```sh
execenv --files <file> [<file> ...] [--missing-env <mode>] [--exec <command> [args...]]
```

Также доступны короткие флаги:

```sh
execenv -f <file> [-m <mode>] [-e <command> [args...]]
```

### Аргументы файлов

Обычный путь к файлу рендерится на месте:

```sh
execenv --files /app/config.yaml
```

Mapping рендерит один файл в другой:

```sh
execenv --files /templates/config.yaml=/app/config.yaml
```

Можно обработать несколько файлов за один запуск:

```sh
execenv --files /app/config.yaml /app/worker.yaml
```

или несколько mapping-аргументов:

```sh
execenv --files \
  /templates/app.yaml=/app/config.yaml \
  /templates/worker.yaml=/app/worker.yaml
```

## Отсутствующие переменные

`--missing-env` управляет тем, что произойдёт, если файл содержит ссылку на переменную окружения, которая не задана.

Доступные режимы:

| Режим | Поведение |
| --- | --- |
| `empty` | Заменить плейсхолдер на пустую строку. Используется по умолчанию. |
| `keep` | Оставить исходный плейсхолдер `${VAR}` без изменений. |
| `error` | Немедленно завершиться с ошибкой и не запускать команду. |

Пример:

```sh
execenv --files /app/config.yaml --missing-env error
```

Для production-окружений обычно безопаснее использовать `--missing-env error`, чтобы приложение не стартовало с неполной конфигурацией.

## Запуск команды

Если передан `--exec`, `execenv` сначала рендерит все указанные файлы. Если рендер успешен, утилита заменяет свой процесс указанной командой через `exec(2)`.

```sh
execenv \
  --files /templates/nginx.conf=/etc/nginx/nginx.conf \
  --missing-env error \
  --exec nginx -g "daemon off;"
```

Всё, что идёт после `--exec`, считается командой и её аргументами, включая значения, которые начинаются с `-`.

Если рендер завершился ошибкой, команда не запускается.

## Пример

Файл-шаблон:

```yaml
server:
  host: ${APP_HOST}
  port: ${APP_PORT}
```

Запуск:

```sh
APP_HOST=0.0.0.0 APP_PORT=8080 \
execenv --files /templates/config.yaml=/app/config.yaml
```

Итоговый файл:

```yaml
server:
  host: 0.0.0.0
  port: 8080
```

## Пример Dockerfile

```dockerfile
FROM chainguard/nginx:latest

COPY target/release/execenv /usr/bin/execenv
COPY templates/nginx.conf /templates/nginx.conf

ENTRYPOINT ["/usr/bin/execenv"]
CMD ["--files", "/templates/nginx.conf=/etc/nginx/nginx.conf", "--missing-env", "error", "--exec", "/usr/sbin/nginx", "-c", "/etc/nginx/nginx.conf", "-e", "/dev/stderr", "-g", "daemon off;"]
```

## Синтаксис плейсхолдеров

Подставляется только форма `${VAR}`.

Примеры:

```text
${DATABASE_URL}
${APP_PORT}
${REDIS_HOST}
```

Другие shell-style формы не раскрываются:

```text
$VAR
${VAR:-default}
${VAR-default}
```

## Замечания по безопасности

`execenv` не вычисляет shell-выражения и не исполняет значения переменных окружения как код. Команда после `--exec` запускается напрямую как программа с аргументами, а не через `sh -c`.

Но если секреты из переменных окружения рендерятся в файлы, их становится проще случайно раскрыть. Будьте осторожены с записью итоговых конфигураций в shared volumes, persistent storage, логи, support bundles или директории, доступные другим процессам.

Рекомендации:

- Считать шаблоны и пути к файлам доверенным вводом.
- По возможности запускать контейнеры не от root.
- Выдавать write-доступ только к директории, где должны создаваться итоговые файлы.
- Использовать `--missing-env error` в production.
- Не записывать секреты в persistent или shared volumes без необходимости.

## Поддерживаемые платформы

`execenv` предназначен для Unix-like систем, потому что `--exec` использует `exec(2)`. Основная целевая среда — Linux-контейнеры.
